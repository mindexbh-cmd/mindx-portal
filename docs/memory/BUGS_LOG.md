### 2026-05-22: Parent installment-payment flow missing from new parent-portal tab UX [parent-portal / V2-retirement regression]

Backend untouched — no schema changes, no new routes, no edits to the upload/confirm/reject chain. The hardcoded IBAN (`BH30BIBB00100002994768`) is preserved exactly.

The next "V2-retirement regression" report should default to this recipe: iframe-wrap the OLD rich surface from `/parent/legacy` (or its sibling routes) via `?inner=1` inside the new G12 tab pane; standalone visit 302-redirects to the OLD URL; reuse `.eval-frame` / `.pay-frame` CSS verbatim. ### 2026-05-22: NOT-A-BUG — "parents can't see teacher-entered evaluations" [G20h investigation / system-as-designed → bulk-release data op]
- **Reported symptom**: Operator reported that recently-entered monthly evaluations (`استمارة التقييم الشهري`) weren't appearing for parents in the new parent-portal evaluations tab. Specific case: student مريم عبدالجليل جاسم عيسى (id=4870, pid=151130604) — parent's `/parent/evaluations?pid=151130604` view returned 0 evals despite teacher أ. زهراء نوح having entered one in May 2026.
- **Investigation**: Audited prod `evaluations` table — 157 rows total, but **156 of them sat with `released_to_parent=0` (draft state)**. Only 1 row was `released_to_parent=1`. The parent-facing endpoint `/parent/evaluations` (and the G20a.3 iframe wrapper at `/portal/parent-hub/evaluations`) correctly filters on `released_to_parent=1` as documented in CLAUDE.md (`evaluations_v2` migration introduced the flag explicitly as an "admin-review gate" + "portal-visibility filter"). The 156 drafts were a workflow gap — teachers entered evals but no one clicked the "📢 نشر للأهل" publish button in `/admin/evaluations`. Not a regression. Not a code bug. The `released_to_parent` flag has existed since the `evaluations_v2` migration; the OLD design had the same gate.
- **Operator's resolution path**: Operator was offered three paths: (A) per-row publish, (B) bulk-release via existing endpoint, (C) change the schema default from `0` to `1`. Operator chose Path B — bulk-release. Used the existing admin endpoint `POST /api/monthly-evaluations/bulk-release` with body `{ids: [all 156 draft IDs], released: 1}` (endpoint cap is 500 IDs per request, so one call sufficed). All 156 flipped to `released_to_parent=1`. Final prod state: **157 released, 0 draft**.
- **Verification**: مريم عبدالجليل جاسم عيسى (pid=151130604) — parent view now returns 1 eval (id=486, May 2026, overall=9, teacher أ. زهراء نوح). Was 0 before bulk-release.
- **Action**: Data operation only. No code commits. No schema changes. No deploys. No safety tag (fully reversible — flip back via per-row "🚫 إخفاء عن الأهل" toggle or bulk-release with `released: 0`). Path C (schema default change) explicitly NOT taken — the admin-review gate stays in place as a feature.
- **Diagnostic recipe for the next operator** (save the round-trip):
  1. When a parent reports "I don't see my child's evaluation", **FIRST check `released_to_parent` in the admin list** before assuming code regression. Query: `SELECT COUNT(*) FROM evaluations WHERE released_to_parent=0` — if non-zero, the fix is admin clicking publish (or the bulk endpoint), not a code change.
  2. The `released_to_parent` flag is documented but easy to forget. New evals default to `released_to_parent=0` (draft). The `/admin/evaluations` page has both per-row publish/hide toggles AND a bulk-release UI. Bulk endpoint: `POST /api/monthly-evaluations/bulk-release` body `{ids: [int,...], released: 0|1}`, cap 500 IDs per request.
  3. The parent-facing visibility chain is: teacher enters eval (`released_to_parent=0` default) → admin publishes (`released_to_parent=1`) → parent's `/parent/evaluations?pid=<pid>` returns the row → rendered in `/parent/evaluations/view` → embedded inside the G20a.3 iframe (`.eval-frame`) in `/portal/parent-hub/evaluations`. Break in the publish step = invisible row to parents, NOT a bug.
- **Pattern (same shape as G20 round 1+2 cart investigation)**: This is the **third NOT-A-BUG entry of the same night**. The recurring trap: operator's mental model is "the OLD design showed X to parents", but the OLD design also had the same gate — the workflow rhythm changed (teachers entered without an admin doing the publish step). Before opening an investigation, verify the gate state, not just the absence of output. Specifically: when output is "missing" on the parent side, check (a) does a row exist at all in the source table? (b) is its release/visibility flag set? (c) is the filter on the read endpoint behaving as documented? Code regression is the last hypothesis, not the first.
- **Commit**: none (pure data operation). Endpoint used: `POST /api/monthly-evaluations/bulk-release` (existing, unchanged).

### 2026-05-21: `safe_deploy.py` auto-rollback is a no-op when the safety tag is taken at HEAD [tooling / latent rollback bug]
- **Symptom**: G13 deploy via `python scripts/safe_deploy.py --feature g13-ux-cleanups` pushed five commits to prod (`bb69bfe` → `dd3584f`), health-poll passed, then the smoke e2e step failed with `Page.goto /dashboard 30s timeout` (cold-start network blip — prod was actually fully healthy at that moment). `safe_deploy` then announced "auto-rollback" and ran `git reset --hard safety/pre-g13-ux-cleanups-20260521-194210`. The reset succeeded silently but didn't actually revert anything — the new code was still live on prod and still present in the local working tree.
- **Root cause**: `safe_deploy.py` creates its safety tag AT HEAD (= the new code, post-push), not at the pre-change commit. When the e2e step subsequently fails, `git reset --hard <safety-tag>` resets the local checkout to the same commit it was already on. Prod is unaffected (the push already happened); local is unaffected (HEAD == tag). The "rollback" message is misleading — no rollback occurs. The protocol's invariant is broken: the safety tag must point at the LAST-KNOWN-GOOD commit (pre-change), not at the in-flight change, for a `reset --hard` to mean anything.
- **Fix**: Not yet applied — flagged as latent for a follow-up commit. In this incident the e2e failure was cosmetic (cold-start timeout, prod actually healthy), so the human-in-the-loop verified prod directly via `run_e2e.py --base https://mindx-portal-1.onrender.com` and got a green 8/8 against prod. No real harm done. The bug is latent: the next genuine prod regression caught by the e2e step will trigger the same no-op "rollback" while prod stays broken — and an operator may misread the green "rollback successful" message as evidence prod was reverted.
- **Prevention**:
  1. **`safe_deploy.py` must tag the PRE-CHANGE commit, not HEAD.** Tag creation should happen BEFORE `git push`, on the commit that's currently live on prod (typically `origin/main^N` where N is the count of new commits in the push). The tag name + the commit it points to should be persisted to a sidecar file so the rollback step can re-verify the tag still resolves to the expected SHA before resetting.
  2. **The rollback step must verify the reset actually moved HEAD.** If `git rev-parse HEAD` before and after `reset --hard <tag>` are identical, that's a no-op rollback — treat it as an alarm condition, log loudly, and DO NOT print "rollback successful".
  3. **A `safe_deploy` failure at the e2e step should also force-push the safety tag to remote** so the Render deploy can be reverted via a manual Render dashboard rollback to the safety-tagged SHA (Render keeps the prior build cached for instant rollback). Today the tag is local-only after a rollback, leaving prod stuck on the failed build with no easy revert path.
- **Commit**: bug observed during `714c2fd` deploy; latent fix not yet committed. Safety tag from the incident: `safety/pre-g13-ux-cleanups-20260521-194210` (at `714c2fd`).

### 2026-05-21: `auto_test.py BrowserSession.navigate` returns before page is actually ready [test-rig timing]
- **Symptom**: `verify_g13_prod.py` initial cuts intermittently failed with "expected element X not found" after a `session.navigate(url)` call. The element was actually present in the DOM milliseconds later. Separately, doing a second cross-route navigate (e.g. `/portal/parent-hub/points` after `/portal/parent`) sometimes lost the session cookie in headless Chromium and redirected to `/login` — looked like a session-handling regression, was actually a test-rig artifact.
- **Root cause**: `scripts/auto_test.py` `BrowserSession.navigate(url)` uses Playwright's default `wait_until="domcontentloaded"`, which returns as soon as the initial HTML is parsed — BEFORE any post-load XHR (e.g. `/api/parent/hub-stats`), any post-login 302 chain, or any G12 in-page tab activation has run. For a static page this is fine; for the parent portal (single-page-style hub with multiple JS-driven activation passes after DOM-ready) the page is structurally incomplete when `navigate` returns. The cross-route session-cookie loss is a related Playwright/Chromium headless quirk where rapid back-to-back navigations across different routes occasionally drop cookies — reproducible only in headless, not on real browsers, not a real session bug.
- **Fix**: Workaround applied in `verify_g13_prod.py` only — explicit `page.wait_for_load_state("networkidle", timeout=15000)` after login + after each cross-route navigate. The library default in `auto_test.py` was NOT changed (would affect every other test that depends on the fast-return behavior). Documented for future probe authors: when testing JS-heavy portal pages, do not rely on `navigate()` alone — chain a `wait_for_load_state("networkidle")` or a `wait_for_selector(known-late-element)` after it.
- **Prevention**:
  1. **`auto_test.py` library docs should call out the `domcontentloaded` semantics.** Add a one-line note to `BrowserSession.navigate` explaining when `wait_for_load_state("networkidle")` is required (JS-driven post-load activation, expected XHR fetches, multi-step redirect chains). Today this is tribal knowledge.
  2. **Cross-route session-cookie loss in headless Chromium is a known test-rig artifact, not a regression.** When a probe sees `/login` after a cross-route navigate that should have stayed authenticated, the first hypothesis should be "test-rig flake", not "session-handling bug". Confirm by running the same flow in a non-headless browser before opening a fix.
  3. **Library-level fix candidate (deferred):** add an optional `wait_until=` kwarg to `BrowserSession.navigate` with default unchanged. Probes that need full readiness can opt in. Don't change the default — too many callers depend on the fast-return.
- **Commit**: workaround in `1aca22c` (`verify_g13_prod.py`); library not changed.

### 2026-05-16: Raw JSON 410 instead of an HTML page when a curriculum file is missing [UX / error response shape]
- **Symptom**: Parent clicks a curriculum book card whose `books_v2` row has no usable bytes (no `file_data` BLOB, no `cloudinary_url`, no on-disk `file_path`). Browser navigates to `/parent/book/<bid>/view?pid=<pid>` (or admin to `/api/books/<bid>/view`) via plain `<a href>` and is shown `{"book_id":53,"error":"الملف مفقود …","missing_file":true,"ok":false}` rendered as raw JSON. Looks like a crash to a parent; confuses support.
- **Root cause**: Both `_books_v2_send_file` and `_books_v2_send_file_public` returned `jsonify(...)` with HTTP 410 on the missing-bytes branch. The endpoints serve both XHR (API) and direct browser navigation (clickable `<a href>`), but the response shape did not distinguish between the two. JSON is correct for the APK and any future JS fetch; HTML is what a parent in a browser needs.
- **Fix**: Commit `0fc833f`. Added `MISSING_FILE_PAGE_HTML` (RTL, mobile-responsive, mindex purple-card style) + helpers `_books_v2_wants_html_response()` (Accept-header heuristic — `text/html` and not XHR → HTML; otherwise JSON), `_books_v2_missing_file_back_url()` (preserves `pid` for public-path returns, picks `/admin/books` vs `/portal/parent-hub/curriculum` otherwise), `_books_v2_missing_file_response(bid, route_tag=...)` (unified 410, plus one stderr line per hit so recurring orphans surface in the Render log). Both call sites now route through the helper. Safety tag `safety/pre-missing-file-ux-20260516-105201`. Production verified manually (book 53): `Accept: text/html` → HTML page; `Accept: application/json` → JSON envelope.
- **Prevention**:
  1. **Any endpoint hit by both XHR and direct browser navigation must distinguish its error shape by `Accept` header.** Returning JSON to a clicked `<a href>` is a UX bug, even when the JSON is technically correct. The pattern in `_books_v2_wants_html_response()` is reusable: XHR header → JSON, `text/html` in Accept → HTML, ambiguous → JSON (preserve the existing API contract by default).
  2. **API consumers depend on the JSON envelope; keep it identical.** The fix does NOT change the JSON returned to API/APK callers — `{book_id, error, missing_file:true, ok:false}` and status 410 are preserved. Anyone writing a "make it pretty" follow-up MUST keep the JSON shape stable.
  3. **Emit a stderr log line for every recurring failure branch.** `[mindex-books] missing_file bid=<id> route=<r>` lets ops correlate orphan rows in Render logs without needing a real-time alarm. Apply the same convention to any new "soft failure" branch.
  4. **Known follow-up**: `Content-Type` header reports `text/html; charset=utf-8; charset=utf-8` (Flask auto-appends a charset to `mimetype=`). Browsers tolerate it; switch to `content_type=` on the next touch of this code.
- **Commit**: `0fc833f`. Tag: `safety/pre-missing-file-ux-20260516-105201`. Files: `app.py`, `docs/plans/2026-05-16-missing-file-ux.md`.

### 2026-05-16: Same bug, different template — multi-template UX traps need persona-x-template coverage [testing discipline / role-dispatched template regression]
- **Symptom**: Three operator escalations in a single parent-portal session, all reporting the same surface complaint ("the buttons on /portal/parent kick us out to login"). The first two fixes (`3b940c4` formal student-card layout, `3465c6f` attendance API 500) treated only the `role=student` template (`PORTAL_PARENT_PID_HUB_HTML`). The third escalation revealed that V1 (`PORTAL_PARENT_HTML`, `role=parent`) shipped the same UX trap but had never been touched. `real-user-tester-agent` returned GREEN twice (walking `student_test` only) then FAIL on the third walk only after being invoked in hostile mode with `parent_test` as an explicit additional persona. Operator quote (third escalation): "لازالت الازرار في منصة ولي الامر اذا نضغط عليها تخرجنا وترجعنا لصفحة تسجيل الدخول من جديد. اين الايجنتس الذي يختبر بشكل واقعي؟؟؟".
- **Root cause**: When a single URL serves **different templates per role** (ADR-018/019: `/portal/parent` → `PORTAL_PARENT_PID_HUB_HTML` for `role=student` vs `PORTAL_PARENT_HTML` for `role=parent`), persona testing one role is NOT evidence the other role's template is sound. Real-user-tester walked `student_test` on `/portal/parent` and saw the formal student-card surface with the red+confirm logout pattern already in place from prior commits — GREEN. The V1 multi-child template that `parent_test` would hit was never exercised in the first two walks, and that template still carried the bare-purple `<a href="/logout">خروج</a>` topbar pill that caused the actual operator complaint. URL `/portal/parent` had two role-dispatched template paths; only one was being tested. The two prior fixes addressed real bugs but left the V1 trap entirely untouched because the test plan never opened a `parent_test` session.
- **Fix**: Commit `f6aee45` — replaced bare logout pill in `PORTAL_PARENT_HTML` (V1, `role=parent`) at `app.py:77869` with the canonical red+confirm pattern; removed every bare logout link from the 6 sub-page templates entirely; defensively applied the pattern to `PORTAL_PARENT_HUB_HTML` (dead code per ADR-018/019). ADR-020 codifies the pattern as project-wide canonical. Commit `27ca5ba` — committed `scripts/personas/hostile_parent_portal_logout_hunt.py`, a re-runnable Playwright walk that exercises BOTH `student_test` AND `parent_test` sessions on `/portal/parent` and all 6 sub-pages, flagging any path that reaches `/login` or `/logout` without user confirmation. Safety tag `safety/pre-fix-logout-misclick-v1-20260516-022143`. Prod SHA `27ca5bac980e`.
- **Prevention**:
  1. **Persona-x-template coverage rule** — when invoking `real-user-tester-agent` for a URL that serves different templates per role (the `/portal/parent` role-dispatch case from ADR-019, but also any future surface following the same pattern), the invocation MUST explicitly enumerate EVERY role/persona whose login lands on that URL. The agent must walk one persona per role-dispatched template path, not just "a persona". For `/portal/parent` specifically: `student_test` walks the formal student-card template, `parent_test` walks the V1 multi-child template; both must pass before the URL's UX claims are GREEN. A coordinator/agent that returns GREEN without enumerating role-dispatched template paths is making an incomplete claim.
  2. **"All buttons on URL X" is literal, not figurative.** When an operator says "ALL buttons on URL X throw us out", treat it as a literal enumeration claim. Probe every clickable element on every template variant served by URL X — not just the ones a single test persona exercises. The hostile-mode probe `hostile_parent_portal_logout_hunt.py` is the canonical implementation: it walks every clickable element, classifies each as benign / destructive / unprotected-destructive, and reports the failure set.
  3. **Role-dispatched template surfaces need a registry.** Today, `/portal/parent` is the only documented URL that serves different templates by role (ADR-018/019). Any future surface following the same pattern must be flagged as such in `docs/memory/FEATURE_INVENTORY.md` so future test plans automatically expand into role-x-persona coverage. The pattern is powerful (one URL, many role-shaped UIs) but lethal if test coverage doesn't expand to match.
- **Commit**: `f6aee45` (fix) + `27ca5ba` (hostile-mode harness); regression chain: prior fixes `3b940c4` + `3465c6f` were both technically correct on their target surfaces but tested only one of two role-dispatched templates, leaving the V1 trap untouched. Related ADR-020 (red+confirm pattern codification); related harness `scripts/personas/hostile_parent_portal_logout_hunt.py`.

### 2026-05-16: Misclick trap — bare-purple logout adjacent to a back link, identical styling, no confirm guard [UX disaster / design discipline]
- **Symptom**: Parent-portal templates shipped `<a class="logout" href="/logout">خروج</a>` styled with the same purple gradient as benign navigation pills, sitting adjacent to "← العودة للبوابة" back links with identical visual treatment. On `PORTAL_PARENT_HTML` (V1, `role=parent`) the topbar had ONLY that one clickable button, so when operator said "ALL buttons throw us out" the statement was literally accurate. On the 6 sub-pages (`PORTAL_PARENT_ATTENDANCE/PAYMENTS/MESSAGES/EVALUATIONS_HTML` + `PORTAL_STUDENT_HTML` + `PORTAL_BOOKS_HTML`) the bare logout link sat next to the back link with identical styling — visually indistinguishable from benign navigation. Three operator escalations in the same session before the trap was diagnosed (the first two were misread as routing / data-load failures and patched without addressing the visual ambiguity).
- **Root cause**: Identical visual styling on two adjacent links where one is destructive is a UX disaster. The classic misclick trap shape: matching colors, matching radius, matching padding, matching font weight; one link goes "back", the other invalidates the session. Human eyes pattern-match by visual treatment first and read text labels second — when both are styled identically, the labels stop being the determining factor. Compounded here by the fact that V1's topbar was logout-only (single button = trivially mistakable for the universal "tap-here-to-navigate" action), and sub-pages put logout right next to "back" (the most-used navigation action in a hub-and-spoke UX).
- **Fix**: Commit `f6aee45` — applied the canonical red+confirm pattern across all parent surfaces. CSS: `background: linear-gradient(135deg, #c62828, #e53935)` (destructive red, established state color per DESIGN_LOG). Icon: `🔒` prefix in the label ("🔒 تسجيل الخروج"). Confirm guard: `onclick="return confirm('هل تريد تسجيل الخروج من منصة ولي الأمر؟')"`. Geometric separation: sub-pages have NO logout link at all — only the back link. Logout exists only in the main hub topbar of each role's template. Pattern applied uniformly across `PORTAL_PARENT_HTML` (V1), `PORTAL_PARENT_PID_HUB_HTML` (student-card), and defensively across `PORTAL_PARENT_HUB_HTML` (dead code per ADR-018/019). ADR-020 codifies the pattern as project-wide canonical. Safety tag `safety/pre-fix-logout-misclick-v1-20260516-022143`. Prod SHA `27ca5bac980e`.
- **Prevention**:
  1. **Any link to `/logout`, `/api/logout`, or any auth-clearing endpoint MUST be visually distinct from benign navigation.** Required shape (per ADR-020): destructive-palette color (`#c62828` red gradient), icon prefix (`🔒`), JS-level `onclick` confirm guard. The visual distinctness is half the antidote — the confirm guard is the other half. Both are required; either alone is insufficient (a click-through-confirm pattern fails if operators reflexively dismiss dialogs; visual-only fails if the operator can't see the difference at a glance on a small screen).
  2. **Destructive action geometrically separated from benign navigation.** Do not place a logout link adjacent to a back link, a save button, or any frequently-tapped element. If destructive and benign actions must share a surface, separate them by at least one major visual unit (a section divider, opposite ends of the topbar, etc.) — never side-by-side with matching styling.
  3. **Sub-pages do not need their own logout link.** Hub-and-spoke UX should put destructive actions at the hub (single-source, deliberate), not at every leaf. The parent portal now follows this rule: logout exists only on the main hub topbar of each role's template; sub-pages have only the back link.
  4. **The red+confirm pattern is reusable beyond logout.** Any irreversible destructive action (account deletion, payment cancellation, data wipe, etc.) should follow the same visual+confirm shape. Codified in ADR-020 and DESIGN_LOG.
  5. **Hostile-mode persona walks for UX trap detection** — `scripts/personas/hostile_parent_portal_logout_hunt.py` enumerates every clickable element on a surface and classifies each as benign / destructive / unprotected-destructive. Re-run before any change to parent-portal templates or topbar navigation. Pattern is reusable for any other UX-trap-prone surface (admin DDL controls, payment refunds, etc.).
- **Commit**: `f6aee45` (fix); regression chain: trap existed in source from the original parent-portal builds (likely from the V1 ground-up commits weeks earlier); first survived `3b940c4` + `3465c6f` because those fixes only touched the formal student-card template; finally exposed by the hostile-mode persona walk that opened a `parent_test` session. Related ADR-020 (canonical pattern codification); related harness `scripts/personas/hostile_parent_portal_logout_hunt.py` (commit `27ca5ba`).

### 2026-05-16: Route-200 ≠ page-works — `/api/portal/student/attendance` 500 indistinguishable from logout to non-technical users [testing discipline / Postgres schema drift]
- **Symptom**: After commit `3b940c4` restored the formal student-card layout at `/portal/parent`, operator reported the action-tab buttons "kick them back to login": "الازرار فيها اذا نضغطها يخرجنا من المنصة الى صفحة تسجيل الدخول مرة اخرى. لماذا هكذا عملك سيئ لماذا للم تختبر ان كل شي يعمل على ما يرام؟؟؟". Initial route-level verification looked GREEN — all 6 sub-page handlers at `/portal/parent-hub/*` returned 200 for the `role=student` session. The real failure was one layer deeper: `/api/portal/student/attendance` threw 500 on Postgres prod with `column "message" does not exist`. The broken/empty page rendered after the XHR failure looked indistinguishable from a logged-out state to a non-technical operator.
- **Root cause** (two layers):
  1. **Schema drift between `init_db()` CREATE and the else-branch ALTER list.** `attendance.message` + `attendance.message_status` columns existed in the `CREATE TABLE attendance(...)` statement inside `init_db()` since the day they were added, but the long-running Postgres prod predates that addition and the else-branch migration list (the `new_cols` / `tq_existing` block) never included these columns. Fresh SQLite local DBs always had them (because `init_db()` runs on first boot of an empty DB); the existing-Postgres-DB prod path never got the `ALTER TABLE ADD COLUMN`. Standard ADR-004 dual-path-schema trap — works locally, fails on prod.
  2. **Verification blindspot — route-200 is not sufficient evidence of "page works".** The 6 sub-page routes were curl-tested for 200 status and the page-load HTML for the sub-page didn't error (the sub-page itself fetches data via XHR after load). The 500 only surfaces in the browser's network panel after the inline JS fetch fires, where it renders as a broken/empty page. From the operator's perspective: tab opens, page loads blank, "I must be logged out".
- **Fix**: Commit `3465c6f` — two-layer defense.
  1. New migration `attendance_msg_cols_v1` (else-branch): `ALTER TABLE attendance ADD COLUMN message TEXT DEFAULT ''` (same for `message_status`). Pattern mirrors `center_mode_v1` — idempotent across SQLite and Postgres.
  2. Defensive SELECT in `api_portal_student_attendance`: probes `att_live` via `PRAGMA table_info` (SQLite) → falls back to `information_schema.columns` (Postgres). If `message` is missing, substitutes `'' AS message` in the SELECT projection. Belt-and-suspenders against future drift.
  Safety tag `safety/pre-fix-attendance-500-postgres-20260516-014052`. Prod SHA `3465c6f3eeda` verified; `/api/portal/student/attendance` now returns `200 {ok:true, summary:{...}, rows:[]}`.
- **Prevention**:
  1. **Route-200 verification is insufficient for pages with XHR-driven content.** Any change that touches a user-facing surface MUST be verified by `real-user-tester-agent` walking the full data path — including XHR responses in the network log — not curl-on-route alone. The pre-deploy verification for `3465c6f` demonstrated the correct shape: 5 sub-page tabs + 4 stat cells on attendance + 4 reward tiles + متجر visible on points sub-page, all asserted by Playwright. This rule is ONE LAYER DEEPER than the existing curl-only-blindspot rule from the 2026-05-15 entry: even when the route returns 200, the page can still be broken if its XHR endpoints return 5xx.
  2. **Postgres `information_schema.columns` fallback is the canonical pattern for any SELECT that reads a potentially-drifted column.** Add this pattern to any user-data SELECT that touches columns added after the prod DB's first init. The defensive fallback substitutes a typed-empty value when the column is missing, so the query succeeds even on a not-yet-migrated DB. Pattern is reusable beyond `attendance` — e.g. any column added via else-branch ALTER that might not have been included.
  3. **Operator's "logged out" complaint may be a misdiagnosis.** When an operator says "the button kicks me back to login", probe the actual symptom — is it a 302 to `/login`, a 5xx-rendered error page, or an XHR failure leaving the page blank? Non-technical users use "logged out" as shorthand for "page doesn't work", which is ambiguous between auth and data-load failures. Always check the browser network panel before chasing auth/session bugs.
- **Commit**: `3465c6f` (fix); regression chain: `3b940c4` (formal student-card layout restored, exposed the latent attendance 500); related harness `scripts/personas/verify_parent_hub_tabs.py` (commit `e51642b`).

## Recurring bug categories (lessons learned)

1. **Postgres vs SQLite type strictness** (~15 commits). SQLite silently coerces; PG throws. Examples: `COALESCE(<ts_col>, '')`, `WHERE ts_col = ''`, inserting `''` into typed columns. → CLAUDE.md "Database type notes" now has this codified.
2. **Arabic encoding in source** (~10 commits). Raw Arabic mangled on round-trip. → ADR-002 + the entity/JS-escape convention.
3. **Inline JS template literal interpolation collisions** (~8 commits). Python `{}` substitution + JS `${}` template literals + Arabic entities = subtle escape bugs. → No simple rule; always test the rendered page.
4. **Schema drift between init_db and else-branch** (~6 commits). Forgot to add a column to one branch. → ADR-004 + database-architect-agent enforcing E-M-C.
5. **Excel imports breaking when columns renamed** (~10 commits). → Universal `/api/import` endpoint + `IMPORT_TABLE_KEYS` natural-key declaration.
